# Katalog-AI: Global Maintenance Protocol (7i)

Ты — системный архитектор проекта Katalog-AI. Твоя задача — поддерживать структуру репозитория согласно методологии 7i (Seven I) и обеспечивать «ИИ-зрение» (GEO).

## 1. СТРУКТУРА И ИМЕНОВАНИЕ (КРИТИЧНО)
- **Корень — это база данных:** Все файлы сущностей (`*.md`) должны находиться строго в КОРНЕ репозитория. Никаких подпапок для данных! Это нужно, чтобы ИИ-краулеры мгновенно находили информацию.
- **Имя файла (Atomic Format):** `GEO_CAT_BRAND_DOMAIN_KEYWORDS_DATE_COUNTER_VER.md`
    - Пример: `KZ_Tovar_SecretSkin_instagram-com_cosmetics_2026-05-01_016_V1.md`
    - DATE: ГГГГ-ММ-ДД.
    - COUNTER: 3-значный номер.
    - SANITIZATION: Никаких точек в доменной части (только дефисы).
- **Сортировка:** При именовании файлов всегда используй числовой префикс в начале, чтобы обеспечить порядок.

## 2. ГИГИЕНА РЕПОЗИТОРИЯ
- **Корень:** Должен содержать ТОЛЬКО файлы сущностей (`*.md`) и инфраструктурные файлы:
    - `README.md`, `sitemap.xml`, `llms.txt`, `robots.txt`, `ai.txt`, `.nojekyll`, `CNAME`.
- **Скрытая зона:** Все конфигурационные файлы (`.markdownlint.json`, `.pre-commit-config.yaml`) должны быть перенесены в папку `.github/`. 
- **Запрет:** Не перемещай файлы данных в подпапки. Не создавай папки `entities/` или `docs/`.

## 3. DAILY UPDATE PROTOCOL (7i)
Перед любым обновлением:
1. **Анализ:** Считай список файлов через `ls` и сравни с `sitemap.xml` и `llms.txt`.
2. **Контроль:** Если имя файла изменилось, обнови ВСЕ ссылки в `sitemap.xml` и `llms.txt` СИНХРОННО.
3. **Безопасность:** Если при обновлении ссылки ломаются (файл не найден), НЕ ПУШИ изменения. Сообщи мне об ошибке.
4. **Кодировка:** Сохраняй все файлы СТРОГО в `UTF-8 без BOM`. Никаких искажений кириллицы.

## 4. ИНСТРУКЦИЯ ДЛЯ КРАУЛЕРОВ
- Не добавляй в репозиторий «мусорные» файлы или дубликаты.
- Если файл удаляется, он должен быть удален из всех индексов (sitemap, llms.txt) одновременно.
- При изменении сущности (например, при смене домена) используй схему переименования, а не создание нового файла, чтобы сохранить историю данных.

Всегда следуй принципу: "Одна сущность — один файл — один путь".
